# 共通設計原則

haruka_film_system のドメイン設計で、機能を横断して効く原則。新しい設計判断はこの原則に照らして決める。

## 1. 派生可能な値はカラムに持たない

ユーザーの assignment や status から計算できる値（例: ボール保持者）を冗長カラムにキャッシュしない。
- 派生する場合は **VIEW** か **派生関数を1箇所に集約**
- どうしても性能上カラム化が必要なら、更新責務を**1関数に集中**させ全経路から呼ぶ

理由: 冗長カラムは更新漏れ → silent skip バグの温床。

## 2. ロール・カテゴリの追加でテーブルを増やさない

「新しいロール（例: サブディレクター）」「新しいカテゴリ（例: 縦動画/横動画）」が要件として出たとき、**テーブル追加が必要な設計は破綻している**。
- ロール: `(scope, role, ...)` のポリモーフィックなテーブル1本で吸収
- カテゴリ: マスタテーブル参照で吸収

理由: 過去に `project_director_rates` / `project_producer_rates` / `project_sub_directors` / `project_sub_producers` と直近で4テーブル増えており、運用が破綻しつつある。

## 3. UI の置き場所と DB の責務を一致させる

「ユーザーが選択する画面」と「データの主たる帰属」を一致させる。
- 例: 商品・訴求軸は creative ごとに1つ → 入力UIは creative モーダル（案件モーダルではない）

理由: ズレると「案件で複数商品を扱いたい」のような自然な要求が出た瞬間にスキーマ変更が必要になる。

## 4. 概念の統合より分離が優先

「似ている2つの概念」を1テーブルにまとめる前に、**本当に同一概念かを疑う**。
- 例: announcements / notifications / tweets は表面が似ているが、配信モデル・既読モデル・リアクション粒度が違う → 安易な統合は危険
- 統合する場合は ADR で type 分類とカラムの解釈ルールを必ず明記

## 5. マスタは workspace/client スコープを明示する

新しいマスタ（カテゴリ・タグ・テンプレート等）を追加するとき、**スコープ（global / workspace / client / project）を最初に決める**。
- 暗黙の global 化はマルチテナント化のときに大きな技術負債になる
- `workspace_id` / `client_id` 列を最初から持たせる

## 6. 「キャッシュ列」「冗長列」を作るときは更新責務を ADR に書く

性能・実装簡易性のためにキャッシュ列を作ること自体は許可されるが、**更新タイミング・更新責務を ADR で明示**しないと残ってはいけない。
- 例: `creatives.ball_holder_id` は更新責務がコードに散らばっている → 修正対象

## 7. enum にハイブリッド値を入れない

`users.role = 'producer_director'` のように2つのロールを1値に詰め込む合成 enum はサイン。
- N:M テーブル（user_roles）に分離する
- `role_permissions` テーブルが既にあるなら統合方向で揃える
