# 018. ディレクション費の課金モード（本あたり / 時給 / 固定）と支払い確定

- **Status**: Accepted
- **Date**: 2026-06-11
- **Decided by**: ユーザー（hiikun.ascs@gmail.com）

## Context

ディレクターへの報酬（ディレクション費）を、誰に・いくら・いつ支払うかを整理する。
意見交換の結果、実態は以下だった。

- ディレクション費には **3 つの課金形態**がある:
  1. **本あたり** … 1 本納品ごとに定額（例: 1,000 円/本）。
  2. **時給** … 実働時間 × 時給（例: 3,000 円/時）。
  3. **固定** … 案件まるごとの定額（例: 30,000 円）。
- ディレクターは「基本は同じ人が最後まで」だが、**途中で離脱・交代する**ことが想定される。
  - 例: 全 3 本のうち 2 本目の途中で交代 → 本 1 は 1 人目、本 2・3 は 2 人目に支払いたい。
  - **途中の本の按分は不要**（仕上げて納品した人が総取り。投げ出した＝未納品分は 0 円）。
- ディレクション費は **成果物グループ（`project_estimate_lines`）に紐づく**（P）。
  「このショート 3 本に対するディレクション費」という単位であり、案件全体から独立した
  統括フィー（Q）ではない、と確認された。

### 現状の仕組みで既に満たされている部分

- `project_estimate_line_costs` は `role_id`（director を含む）× `user_id` ×
  `pricing_type`（`fixed_per_unit` / `percentage` / `hourly` / `fixed_total`）を持つ。
  → **3 モードのデータ表現は既存スキーマで可能**。UI にも本あたり / 時給 / 固定 /
  パーセンテージのラジオと実工数入力欄が実装済み
  ([public/haruka.html:5660](../../../public/haruka.html#L5660),
  [:5685](../../../public/haruka.html#L5685))。
- `UNIQUE (line_id, role_id, user_id)` のため、**同一成果物グループに
  「2 人目のディレクター行」を user_id 別に追加できる**。

### 足りない部分

- **本あたりモードの途中交代**は、納品時に担当ディレクターを固定しないと
  `projects.director_id` の書き換えで過去の納品分まで付け替わる
  （= ADR 009 が「バグに近い設計」と認定した問題）。**ADR 009 は Accepted だが未実装**。
- **固定 / 時給の支払い確定タイミング**が未定義（本数に直結しないため、納品トリガーで
  確定できない）。

## Decision

ディレクション費は **成果物グループの `project_estimate_line_costs`（role=director）に
持つ**。`pricing_type` でモードを選ぶ。帰属・交代・確定の扱いはモードごとに以下とする。

### モード別の挙動

| モード | pricing_type | 計算 | 確定トリガー | 途中交代の割れ方 |
|---|---|---|---|---|
| **本あたり** | `fixed_per_unit` | 単価 × 納品本数 | **本の納品完了**（本ごと） | **本ごとに自動**（納品時にその本のディレクターを固定。ADR 009） |
| **時給** | `hourly` | 時給 × 各人の実働時間 | **案件完了** | **人別行で自動**（田中の時間・鈴木の時間を別行で入力） |
| **固定** | `fixed_total` | 人別の固定額 | **案件完了** | **人別行で手動分割**（「田中分 1 万」「鈴木分 2 万」を別行で入力） |

- **按分ロジックは持たない**。本あたりは「納品した人が総取り」、固定は「合意額を
  admin が人別に入力」、時給は「各人が自分の実働時間を入力」で、いずれも按分計算は不要。
- 途中交代は **同一 line に user_id 別の director 行を複数立てる**ことで表現する
  （本あたりのみ、本数の帰属を line_cost 行ではなく ADR 009 のスナップショットで解決）。

### 本あたりモードの交代 ＝ ADR 009 に委譲

本あたりの「納品時に本ごとのディレクターを固定する」挙動は **ADR 009（納品スナップショット）
に全面的に依存する**。本 ADR は ADR 009 の実装を前提とし、重複定義しない。

- ADR 009 が未実装の現状では、本あたりモードで途中交代すると過去の納品分が付け替わる。
  → **本 ADR を実装に移す際は、ADR 009 のスナップショット実装を本あたりモードの前提条件とする。**

### 固定 / 時給の支払い確定 ＝ 案件完了トリガー

固定・時給は本数に直結しないため、**案件が「完了」ステータスに遷移した時点で確定**する。

- 本あたりが creative 単位（納品時）に確定・請求経路へ乗るのに対し、
  固定 / 時給のディレクション費は **creative 単位の請求には乗せず、案件完了時に
  1 回だけ確定する別集計**とする。
- 確定時、その line の director 行（user_id 別、`hourly` / `fixed_total`）を集計し、
  各人への支払い額を確定する。
  - `hourly`: `unit_price × actual_hours`（各行が持つ実働時間）。
  - `fixed_total`: `unit_price`（人別に入力された固定額をそのまま。**本数で割らない**）。

### fixed_total の本数按分を行わない

現行 `resolveCreativeRoleCost` は `fixed_total` を `÷ planned_count` で creative 単位に
按分する。これは「案件まるごと固定費を creatives へ分配して請求する」用途の挙動。
ディレクション費の固定モードでは、**計画本数と実納品本数のズレで総額が合わなくなる**
（計画 10 本 / 実績 12 本なら 2 割過払い）。

→ **ディレクション費（role=director）の `fixed_total` は本数按分せず、案件完了時に
人別固定額を満額 1 回計上する**。pricing.js の per-creative 按分経路とは分離する。

## Consequences

### Pros
- 3 モードすべてが既存スキーマ（line_costs の pricing_type + user_id 複数行）に乗る。
  **新規テーブルは不要**。
- 途中交代が、本あたり＝自動（ADR 009）、時給＝実働時間で自動、固定＝人別手動入力、と
  いずれも按分計算なしで合計が必ず合う。
- 「投げ出した＝未納品分は 0 円」が本あたりで自然に成立。

### Cons / Trade-offs
- 本あたりモードは **ADR 009 の実装完了が前提**。ADR 009 未実装のままでは交代時の
  正しさを保証できない（実装順序の制約）。
- 固定 / 時給は creative 単位の請求経路とは別に「案件完了時確定」の集計経路を持つため、
  集計ロジックが本あたりと二経路になる。
- 固定 / 時給で途中交代した場合、行の追加・実働時間入力は **admin の手動運用**に依存する
  （自動分割ではない）。

### 残課題（実装時に詰める）
- **本あたりの再納品付け替え**: 一度納品 → 差し戻し → 別人が再納品したケースで、
  ADR 009 は「最初のスナップショットを上書きしない」ため旧担当に残る。
  admin 用の手動再スナップショットを認めるかは ADR 009 Phase 2 の射程（本 ADR では未確定）。
- **案件完了の定義**: 固定 / 時給の確定トリガーとなる「案件完了」が、どのステータス /
  操作を指すか（`projects.status` の値、admin の明示操作の有無）を実装時に確定する。
- **外部ディレクター**（ADR 017, `is_external`）に director_fee を立てるかの扱い。

## Alternatives Considered

### A. 固定 / 時給を「案件まるごと（Q）」の固定費欄（project_fixed_items）で持つ
- ディレクション費が本数非依存なら `project_fixed_items`（ADR 006）が自然な置き場所。
- ❌ 実態は成果物グループ単位（P）であり、案件全体から独立した統括フィーではないと
  確認された。成果物グループに紐づけた方が「このショート 3 本のディレクション費」という
  単位に合う。Q が出てきたら別途 ADR 006 で対応する。

### B. 固定 / 時給も creative 単位（納品時）に按分して請求経路に乗せる
- 本あたりと経路を統一できる。
- ❌ `fixed_total ÷ planned_count` は実納品本数とズレると総額が合わない。時給も
  「各本に何時間」を割り当てる必要が出て過剰。固定 / 時給は本来「本数に直結しない」性質
  なので、案件完了時の一括確定が素直。

### C. 途中交代に按分機能を標準実装する（工程到達率で自動分割）
- 1 本の途中交代でも公平に分配できる。
- ❌ ユーザーは「按分なし・総取りで OK」と明言。工程到達率の計測・支払いトリガーの爆発を
  招く割に、稀ケースのための過剰設計。手動分割（固定は人別行、本あたりは creative_cost_overrides
  / ADR 013）で足りる。

## Migration / Rollout

1. **前提**: ADR 009（納品スナップショット）の実装。本あたりモードの交代正しさはこれに依存。
2. **Phase 1**: 固定 / 時給のディレクション費を案件完了時に確定する集計経路の追加。
   director 行（user_id 別 `hourly` / `fixed_total`）を案件完了時に集計。
   `fixed_total` は本数按分しない分岐を pricing.js に明示。
3. **Phase 2 (将来)**: 案件完了確定の UI（admin が確定額を確認 / 実働時間を入力）。
