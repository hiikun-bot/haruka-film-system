# ADR 023: 素材広場の任意フォルダ（クライアント→案件 直下 / ルート直下）

- Status: Accepted
- Date: 2026-06-23
- Related: ADR 018（WebP プレビュー経由解析）, ADR 019（解析完了で自動振り分け）, ADR 020（プレビュー）, `migrations/2026-06-16_msquare_project_id.sql`（案件紐付け）

## Context

素材広場では「クライアント → 案件」の 2 階層までは
`lib/video-organization/project-folder.js` の `resolveProjectFolder()` が
Drive 上に ensure し、アップロード／自動振り分けの両方で同じ案件フォルダ直下に
素材を集約している（ADR 019 + project_id 化）。

しかし実運用では、1 つの案件の中で **素材を撮影回・用途ごとにまとめたい** という
ニーズがある。例えば同じ案件で

- 6/12 に撮った Zoom 録画
- 6/13 に撮った対談動画

を別々の箱に分け、「この箱の素材からこのリールを作る」という単位で扱いたい。
現状は案件フォルダ直下にファイル名（先頭 `[YYYYMMDD]`）でフラットに並ぶだけで、
画面上にも「箱」の概念が無い。

要件（ユーザー指示）:

1. この画面でフォルダを作ったら Google Drive 上にも同じフォルダを同期する。
2. フォルダは **クライアント → 案件 の直下** に作成できる。クライアント→案件まで
   選択されていることを必須とする（案件の下にぶら下げる箱だから）。
3. クライアントを選ばないケースに限り、**素材広場ルート直下** にフォルダを作成できる。
4. 作ったフォルダへ素材を移動（振り分け）でき、画面でもフォルダ単位に絞り込める。

## Decision

### フォルダの実体と階層

素材広場の「任意フォルダ」を `material_square_folders` テーブルで管理する。
Drive 上の階層は既存の命名・サニタイズ規則（`sanitizeFolderName`）をそのまま使い、

```
素材広場ルート（VIDEO_ORG_UPLOAD_FOLDER_ID）
  └─ クライアント名                ← resolveProjectFolder が ensure
      └─ 案件名                    ← resolveProjectFolder が ensure
          └─ [任意フォルダ名]      ← 本 ADR で追加（project_id あり）

素材広場ルート
  └─ [任意フォルダ名]              ← 本 ADR で追加（client/project なし）
```

の 2 パターンのみを許可する。**「クライアントは選んだが案件は未選択」という
中途半端な階層には作らせない**（要件 2/3）。これを DB の CHECK 制約と API
バリデーションの両方で担保する。

### なぜ Drive 上だけでなく DB でも持つか

- 画面でフォルダ一覧（空フォルダ含む）・素材件数バッジ・絞り込みを出すため、
  Drive を都度 list するのではなく DB を正とする（Drive list は Shared Drive で
  レイテンシ・ページングが重く、空フォルダ判定もしづらい）。
- 素材 → フォルダの所属は `video_file_organization_tests.folder_id` で表現し、
  既存の一覧クエリ（`GET /list`）に `folderId` フィルタを 1 本足すだけで
  「このフォルダの素材だけ」を実現できる。

`folder_id` は **案件（project_id）とは独立**。フォルダ自身が project_id を持つので、
素材の project_id とフォルダの project_id は整合しているのが正常系だが、
不整合（別案件のフォルダへ移動）は API 側で弾く（後述）。

### Drive 操作はすべて SA（サービスアカウント）

ADR 019 改訂（2026-06-16）で、素材広場の Drive 操作は user OAuth を撤回し
SA に統一済み。本 ADR のフォルダ作成・素材移動も `driveLib.getDriveService()`
（SA）/ `getOrCreateFolder()` を使う。SA は実運用上すでに共有ドライブのメンバー。

### 削除の扱い（SA は完全削除不可）

ADR と整合（素材市場の削除はゴミ箱送り必須＝SA は共有ドライブで `canDelete=false`）。
フォルダ削除は **中に素材が無い（folder_id 参照が 0 件）の空フォルダのみ許可** し、
Drive 上は `trashFile()`（ゴミ箱送り）で処理する。中身があるフォルダは
「先に素材を移動／フォルダ解除してください」と促し、削除させない。これにより
孤児ファイル（PR #826 で問題化した 11GB 相当）を新たに生まないようにする。

### API（`routes/video-organization-test.js`）

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/folders?clientId=&projectId=` | フォルダ一覧 + 各フォルダの素材件数。`projectId` 指定でその案件直下、無指定（client/project とも無し）でルート直下。`clientId` のみ（案件未指定）はバリデーションエラー扱いにせず「ルート直下扱い」ではなく **400 を返す前にフロントで案件選択を促す**運用とし、API は `projectId` 有り or 完全無指定の 2 系統で受ける |
| POST | `/folders` | `{ name, projectId? }`。`projectId` 有り → `resolveProjectFolder` で案件フォルダを ensure → その直下に `getOrCreateFolder`。無し → ルート直下に `getOrCreateFolder`。Drive folderId と client_id/project_id を DB に INSERT |
| POST | `/folders/move` | `{ fileIds:[], folderId|null }`。各素材を Drive 上で対象フォルダへ `files.update(addParents/removeParents)` 移動し、`folder_id` を更新。`folderId=null` で「フォルダ解除＝案件フォルダ（or ルート）直下へ戻す」 |
| DELETE | `/folders/:id` | 空フォルダのみ。Drive を `trashFile` し DB から削除 |

`POST /folders` のバリデーション:

- `projectId` 有り: 案件が存在し、その `client_id` を `material_square_folders.client_id`
  へ写す（一覧の高速フィルタ用に非正規化）。
- `projectId` 無し: client/project とも NULL でルート直下。
- 同一階層・同一名は `getOrCreateFolder` が冪等に再利用（重複行は作らないよう
  既存 DB 行も name + 階層キーで UPSERT 的に扱う）。

`POST /folders/move` のバリデーション:

- 移動先フォルダの `project_id` と素材の `project_id` が異なる場合は弾く
  （案件をまたいだ移動は Drive 階層が破綻するため）。`folderId=null`（解除）は常に許可。

### フロント（`public/haruka.html`）

一覧ビュー（`#ms-view-list`）のクライアント／案件フィルタ直下にフォルダ行を追加する。

```
┌─ 素材広場 ─────────────────────────────────────────────┐
│ 🔍[検索] 状態[すべて▼] 種別[すべて▼] クライアント[ABC社▼] 案件[夏CP▼] │
│ ─────────────────────────────────────────────────────── │
│ 📁 フォルダ：  [すべて] [6/12 Zoom (3)] [6/13 対談 (5)]  ＋フォルダを作成 │
└─────────────────────────────────────────────────────────┘
```

- 「＋フォルダを作成」は **クライアント＋案件を選択済み**、または
  **クライアント未選択（＝ルート直下）** のときだけ活性。
  クライアントのみ選択（案件未選択）のときは非活性＋
  「案件まで選ぶとその直下に、クライアント未選択なら素材広場直下に作成できます」
  のヒントを出す（要件 2/3 の UI ガード）。
- フォルダチップをクリックすると `GET /list?...&folderId=` で絞り込み。
  もう一度押すと解除（[すべて]）。
- 複数選択バーに「📁 フォルダへ移動」を追加。移動先フォルダ選択ポップオーバーに
  「フォルダから外す」を含める。

作成モーダル（絵コンテ）:

```
┌─ フォルダを作成 ──────────────────────────┐
│ 作成先：ABC社 ／ 夏キャンペーン2026         │  ← project 選択時。未選択なら「素材広場（直下）」
│ フォルダ名：[ 6/12 Zoom動画            ]   │
│            ＼/ ? % * : | " < > は _ に変換  │
│                          [キャンセル] [作成] │
└────────────────────────────────────────────┘
```

## Consequences

### Positive
- 案件の中を撮影回・用途で箱分けでき、「この箱からこのリール」という制作単位が画面で表現される。
- Drive 側にも同じ箱ができるので、編集者が Drive を直接開いても整理されている。
- 既存の `resolveProjectFolder` / `getOrCreateFolder` / `GET /list` を最小拡張で流用でき、新しい Drive 同期経路を増やさない。

### Negative / Risks
- 素材の `folder_id` と所属案件（`project_id`）の二重管理になる。案件をまたぐ移動を API で弾いて整合を守る。
- フォルダ名のサニタイズで別名 2 つが同一 Drive 名に潰れる可能性（既存仕様と同じ）。`getOrCreateFolder` の冪等性で同一フォルダに収束する。
- 自動振り分け（auto-apply）は引き続き「案件フォルダ直下」に置く。フォルダへ入れるのは手動移動のみ（将来 AI 提案でフォルダまで割り当てる余地は open）。

## Alternatives Considered

- **Drive 上にだけフォルダを作り DB を持たない案**: 実装は軽いが、画面でフォルダ一覧・件数・絞り込み・空フォルダ表示ができない（要件「画面にフォルダ一覧も表示」を満たせない）。却下。
- **projects テーブルにサブ案件として持つ案**: フォルダは見積・単価・納期を持たない単なる箱であり、案件のライフサイクル（ADR 005）に乗せると概念が濁る。独立テーブルにする。
- **クライアントのみ選択でクライアント直下に作る案**: 階層が「クライアント直下 / 案件直下 / ルート直下」の 3 系統に増え、素材の所属がブレる。要件どおり「案件直下 or ルート直下」の 2 系統に限定する。
