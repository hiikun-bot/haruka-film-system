# 017. ロール別ダッシュボード：自分の仕事 × 一つ上の視座 × モチベーション数値

- **Status**: Proposed
- **Date**: 2026-05-11
- **Decided by**: ユーザー（hiikun.ascs@gmail.com）

## Context

現状のホーム画面（[public/haruka.html#page-home](../../../public/haruka.html#L3587)）は事実上 **管理層 / 制作層の二項対立** でしか出し分けされていない。

```
if (['admin','secretary','producer','producer_director'].includes(role)) {
  renderAdminDash(today);   // 4種類: 経営チャート＋全体俯瞰
} else if (role === 'director') {
  renderAdminDash(today);   // ↑と同じ
} else {
  renderEditorDash(today);  // editor/designer: 自分の担当のみ
}
```
（[public/haruka.html:17756-17762](../../../public/haruka.html#L17756)）

### 浮上した課題

| ロール | 現状の問題 |
|---|---|
| `secretary` | admin と同じ売上予実が出る一方、本業の請求書・支払い管理ブロックが無い |
| `director` | producer と同じ全社俯瞰が出るが、本人が見たいのは「自分の案件」だけ |
| `producer` / `producer_director` | 自分の担当案件にフォーカスした粗利・ボール状態の俯瞰が無い |
| `sub_producer` / `sub_director` | 明示分岐なし。フォールバックで editor 扱いに落ちている可能性が高い |
| `editor` / `designer` | タスク列挙のみ。「累計実績」「成長指標」などモチベーションを刺す数値が一切無い |

### 設計思想（ユーザー判断）

ロール別ダッシュボードを設計するにあたり、以下3点をすべての画面で同居させる。

1. **自分の仕事を全うする**ブロック（タスク・ボール・納期）
2. **一つ上の役職の視座を覗ける**ブロック
   - キャリア形成と主体性の促進が目的
   - 「上司は今こういう数字に追われている」が見えると、現場の判断が一段階成熟する
3. **モチベーションとなる数値**
   - 累計納品本数、リピート率、回収率など「追いかけたくなる数字」
   - 「数値を追い求める人にとっては有益な情報」を提供する

## Decision

### ロールヒエラルキーと「一つ上」マッピング

| 自ロール | 一つ上として覗ける視座 |
|---|---|
| `editor` / `designer` | `sub_director`（同案件のディレクション全体） |
| `sub_director` | `director`（案件のクリエイティブ全体） |
| `director` | `producer`（案件 P&L・受注見込み） |
| `sub_producer` | `producer` |
| `producer` / `producer_director` | `admin`（経営：全社売上・粗利） |
| `secretary` | `admin`（経営：キャッシュフロー視点） |
| `admin` | （最上位。代わりに「未来予測」を覗く） |

**実装上の注意**：覗き視座ブロックは閲覧専用。アクション（編集・承認）は元のロール権限に従う。VIEW AS との混同を避けるため、ブロックタイトルに「↑ 一つ上の視座」アイコンを付ける。

### ロール別ダッシュボード構成（マトリクス）

#### 共通ヘッダー（全ロール）
- 挨拶 / 日付 / 名言
- お知らせ（`announcements`）
- 誕生日（`users.birthday`）

#### admin（最高経営責任者視点）

```
┌─ ヒーロー: 進行中案件 / 今月本数 / 納品済 / 売上見込 ─┐
├─ ① 全体納期アラート（遅延・今日・3日以内）             │
├─ ② 売上予実 ＋ 月別納品本数 チャート                   │
├─ ③ 案件ステータス円 ＋ クリエイティブステータス円       │
├─ ④【未来予測】3ヶ月先の受注見込み・粗利予測             │← 覗き枠相当
├─ ⑤【モチベ数値】年間累計売上 / 前年同期比 / 過去最高月  │
└────────────────────────────────────────────────────┘
```

#### secretary（請求・支払い専門 ※前提：本業は経理ライン）

```
┌─ ヒーロー: 今月請求件数 / 未送付 / 入金待ち / 支払予定 ─┐
├─ ① 請求書アラート（今月発行予定・未送付・入金遅延）   │
├─ ② 支払い予定（外部スタッフへの今月支払い一覧）       │
├─ ③ 全体納期（読み取り専用、請求タイミング把握用）     │
├─ ④【↑ 一つ上：admin 視座】今月の売上予実ミニカード     │
├─ ⑤【モチベ数値】今月処理済請求書数 / 回収率 / 平均回収日数│
└────────────────────────────────────────────────────┘
```

#### producer / producer_director（案件統括）

```
┌─ ヒーロー: 担当案件数 / 今月納品本数 / 自分ボール / 遅延 ─┐
├─ ① 自分がボール保持の案件・クリエイティブ                │
├─ ② 担当案件の今週の山場（マイルストーン）                │
├─ ③ 担当案件の粗利ランキング（上位5件）                   │
├─ ④【↑ 一つ上：admin 視座】全社売上予実ミニカード          │
├─ ⑤【モチベ数値】年間累計受注額 / リピート受注率 / 平均粗利率│
└────────────────────────────────────────────────────┘
```

#### sub_producer（プロデューサー補佐）

```
┌─ ヒーロー: 担当案件数 / 自分ボール / 今週納期 / 遅延 ─┐
├─ ① 自分がボール保持のタスク                          │
├─ ② 担当案件の今週の山場                              │
├─ ③【↑ 一つ上：producer 視座】担当案件の粗利ランキング │
├─ ④【モチベ数値】今月の関与案件数 / クライアント満足度  │
└────────────────────────────────────────────────────┘
```

#### director / sub_director（制作進行）

```
┌─ ヒーロー: 担当クリエイティブ数 / 自分ボール / 今週納期 ─┐
├─ ① 自分がボール保持の作業（最優先）                     │
├─ ② 担当案件の今週の山場                                │
├─ ③ 今日のタスク                                        │
├─ ④【↑ 一つ上：producer 視座 / director の場合は admin】 │
│     担当案件の粗利ミニカード / 全社売上予実              │
├─ ⑤【モチベ数値】年間累計担当本数 / 1発OK率 / 平均修正回数│
└────────────────────────────────────────────────────┘
```

#### editor / designer（実作業）

```
┌─ ヒーロー: 担当本数 / 今週納期 / 自分ボール ─┐
├─ ① 自分のタスクキュー（ボール来てる順 ※ADR 016 連携）│
├─ ② 今週の納期（自分の担当のみ）                     │
├─ ③ 修正回数の多い案件アラート（学びのタネ）          │
├─ ④【↑ 一つ上：sub_director 視座】案件全体の進捗ミニ表 │
├─ ⑤【モチベ数値】年間累計納品本数 / 最高月本数 /      │
│     自分の作品がリリースされた回数（version_logs連携） │
└────────────────────────────────────────────────────┘
```

### モチベーション数値の設計指針

「追いかけたくなる数字」を以下4軸で揃える：

| 軸 | 例 |
|---|---|
| **累積（誇り）** | 年間累計納品本数、累計受注額、累計回収額 |
| **率（クオリティ）** | 1発OK率、リピート率、回収率、粗利率 |
| **ピーク（記録更新）** | 過去最高月本数、過去最高粗利月 |
| **貢献（社会的承認）** | リリース回数、Verup情報の登場回数、クライアントからの感謝コメント数 |

### 「一つ上の視座」ブロックの実装制約

- 既存の「`hasPermission(key)` での権限チェック」とは別レイヤー
- 覗き視座ブロックは **集計値のみ・個別レコード非表示**（権限漏洩防止）
- ブロック内に「もっと詳しく → VIEW AS で切り替え」リンク（admin のみ表示）
- 表示ロジックは `effectiveRole()` ベース（ADR 015 準拠）

## Consequences

### Positive
- ロール本来の役割が画面に反映され、「最初に何を見るべきか」が明確になる
- 「一つ上の視座」が現場の主体性を促進し、キャリア形成と意思決定の質が上がる
- モチベーション数値により、編集者・デザイナーが「数字で評価される」実感を持てる
- secretary が請求業務に集中できる（admin と同じ画面を出さない）

### Negative
- ダッシュボードの実装量が増える（5パターン × ブロック単位）
- モチベ数値（累計・最高記録）の集計クエリは重い → マテリアライズドビューや日次バッチが必要になる可能性
- 「一つ上の視座」の集計が誤って詳細レコードを漏らさないよう、API レイヤーでの集計強制が必要

### Migration / Rollout

ステージ分割で段階リリース：

| Stage | 内容 |
|---|---|
| **Stage 1** | secretary 専用ダッシュボード（請求・支払いブロック）。最も乖離が大きい |
| **Stage 2** | editor / designer のモチベ数値（年間累計・1発OK率）+ ボール来てる順並び替え |
| **Stage 3** | producer / director の「自分の案件」フォーカス + 粗利ランキング |
| **Stage 4** | 「↑ 一つ上の視座」ブロックを全ロールに展開 |
| **Stage 5** | admin の「未来予測」ブロック（受注見込み・粗利予測） |

各 Stage で集計に必要な DB ビュー / インデックス / 集計バッチを同 PR で追加する。

## Alternatives Considered

### A案: VIEW AS をもっと強化して、admin が任意ロールを切り替えればいい

**却下理由**：VIEW AS は admin の確認用機能。一般メンバーには使えない。各メンバー本人にとってのダッシュボードを設計する目的とは別レイヤー。

### B案: ダッシュボードはやめて「タスク一覧」だけ出す（Linear / Asana 風）

**却下理由**：「数値で追い求めたい人へのワクワクを提供する」というユーザー要望と矛盾。タスク管理ツールではなく、組織の状態を映すホームを作る。

### C案: ロール別ではなく、ユーザー個別にウィジェット選択させる（カスタマイズ式）

**却下理由**：自由度が高すぎて誰も整理しない。ロール別の「推奨レイアウト」を提供したほうが組織の標準が育つ。Stage 5 以降の拡張余地として残す。

## Open Questions

- `version_logs` の登場回数集計を「自分の作品がリリースされた回数」として使うのは妥当か？ → `version_logs.creative_id` のリレーションが必要か検討
- 「1発OK率」は `creative_versions.round_number = 0` で納品確定したクリエイティブ数 / 総納品数 でよいか？
- secretary が「全体納期」を読み取り専用で見る場合、どこまでの詳細（金額・担当者）を見せるか？
- 「↑ 一つ上の視座」ブロックは折りたたみ式 vs 常時展開どちらがよいか？（モバイル考慮）
