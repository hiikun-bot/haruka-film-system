# 画面別 潜在的不具合レビュー（簡易静的調査）

対象:
- `public/login.html`
- `public/invite.html`
- `public/haruka.html`
- `public/editor.html`

## 1) ログイン画面 (`login.html`)

- **多重送信リスク（中）**
  - `login()` 実行時にボタン表示はローディング化されるが、`disabled` 制御が見当たらず、Enter 連打やダブルクリックで複数リクエストが飛ぶ可能性。
- **エラーハンドリングの粒度不足（低）**
  - 失敗時メッセージの詳細分岐が薄い場合、認証失敗・ネットワーク断・サーバーエラーが同じ見え方になり、ユーザーが再試行判断しづらい。

## 2) 招待登録画面 (`invite.html`)

- **トークン検証失敗時のUX（低〜中）**
  - 期限切れ/無効トークン時に再発行導線が不足すると離脱につながる。
- **登録ボタンの再入防止（中）**
  - ローディング表示変更はあるが、`disabled` 付与がないと多重登録API呼び出しの恐れ。

## 3) メイン画面 (`haruka.html`)

- **`innerHTML` 動的描画のXSS/DOM破壊リスク（高）**
  - 大量の画面領域で `innerHTML` を使っており、データ由来の文字列にエスケープ漏れがあるとXSS化しやすい。
  - 一部は `esc()` 利用が見えるが、全経路で徹底されているか点検が必要。
- **インライン `onclick` 依存（中）**
  - 動的テンプレート内に `onclick` 文字列連結が多く、クォート混入やID値の予期しない文字でイベントが壊れる可能性。
- **グローバル状態肥大化（中）**
  - 単一HTMLに機能が集約され、ページ間状態やタイマー/イベント解除漏れで副作用が出やすい（モーダル切替、動画プレイヤー、集計画面など）。
- **`setInterval`/`addEventListener` の解除漏れ（中〜高）**
  - プレイヤー系や監視処理で繰り返し登録箇所があり、再描画時の二重登録・メモリ増加・多重実行が起きうる。
- **新規タブ遷移時のフォールバック不足（低）**
  - `window.open(...,'_blank','noopener')` は概ね安全だが、ポップアップブロック時の代替導線が弱い。

## 4) エディター画面 (`editor.html`)

- **コメント描画の `innerHTML` 依存（中〜高）**
  - コメント本文やユーザー名の描画でエスケープ漏れがあるとXSSリスク。
- **ポーリングの停止条件（中）**
  - `setInterval` 利用時に画面離脱/アセット切替で停止漏れがあると二重ポーリング化しやすい。
- **再解析/削除など破壊操作の誤操作防止（低〜中）**
  - 破壊系操作に confirm・連打抑止・楽観更新のロールバックを統一実装しておくと事故が減る。

## 優先度つき改善提案

1. **最優先**: `innerHTML` の全面棚卸し。
   - 原則 `textContent` / `createElement` 化。
   - どうしてもHTMLテンプレートが必要な箇所は共通 `escapeHtml()` を強制。
2. **高**: API送信ボタンの共通多重送信ガード。
   - 送信開始で `disabled=true`、finally で復帰。
   - Enter連打も吸収できるフォーム送信ガードを追加。
3. **高**: タイマー・イベントのライフサイクル管理。
   - 画面遷移/モーダルclose時に `clearInterval` / `removeEventListener` を必ず実行。
4. **中**: インラインイベントを委譲方式に置換。
   - `addEventListener` + `data-*` へ移行し、文字列連結バグを削減。
5. **中**: エラーメッセージ標準化。
   - `network / auth / validation / server` をUIで判別表示。

## すぐできる確認観点（手動テスト）

- ログイン/招待登録でダブルクリック・Enter連打してAPI呼び出し回数を確認。
- メイン画面でモーダルを開閉し続け、同一イベントが複数発火しないか確認。
- エディターでアセット切替を連続実行し、ポーリング回数が増殖しないか確認。
- コメントや名称に `"'<>` を含む文字を入力して崩れ/XSSが発生しないか確認。
